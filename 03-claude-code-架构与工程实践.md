# Claude Code 架构与工程实践

## 前言：氪金换来的教训

这篇文章源自作者深度使用 Claude Code 半年、**两个账号每月 40 刀**换来的血泪经验。刚开始把 Claude Code 当 ChatBot 用，结果发现：

- 上下文越来越乱
- 工具越堆越多效果越差
- 规则越写越长越不遵守

折腾一番后发现：**这不是 Prompt 问题，而是这套系统的设计就是这样的**。

---

## 六层架构：Claude Code 的真实面目

Claude Code 不是简单的"问答"系统，而是一个**反复循环的代理过程**：

```
    ┌─────────────────────────────────────────┐
    │                                         │
    │   ┌─────────┐    ┌─────────┐    ┌─────────┐  │
    │   │ 收集    │ →  │ 采取    │ →  │ 验证    │  │
    │   │ 上下文  │    │ 行动    │    │ 结果    │  │
    │   └─────────┘    └─────────┘    └─────────┘  │
    │       ↑                              │      │
    │       │            完成或重新开始     │      │
    │       └──────────────────────────────┘      │
    │                                             │
    │  ┌─────────────────────────────────────┐   │
    │  │  CLAUDE.md | Hooks | 权限 | 沙箱      │   │
    │  │  Skills    | Tools    | MCP          │   │
    │  │  Memory                           │   │
    │  └─────────────────────────────────────┘   │
    └─────────────────────────────────────────┘
```

### 关键认知：只强化一层，系统就会失衡

```
┌────────────────────────────────────────────────────────────┐
│                                                            │
│  CLAUDE.md 写太长 → 上下文先污染自己了                      │
│  工具堆太多 → 选择就搞不清楚了                              │
│  subagents 到处开 → 状态就漂移了                            │
│  跳过验证环节 → 出问题了都不知道是哪里挂的                   │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

说实话，卡住的地方几乎从来不是模型不够聪明，更多时候是：
- 给了它错误的上下文
- 或者写出来了但根本没法判断对不对
- 也没法撤回

## 上下文为什么会乱：不是容量问题，是太吵了

很多人以为上下文是"容量问题"，但实际是"太吵了"——有用信息被大量无关内容淹没。

### 200K 上下文的真实开销

```
┌─────────────────────────────────────────────────────────────┐
│  200K 总上下文                                               │
├─────────────────────────────────────────────────────────────┤
│  固定开销 (~15-20K)                     ← 这部分偷偷吃掉很多 │
│  ├── 系统指令: ~2K                                           │
│  ├── Skill 描述符: ~1-5K                                     │
│  ├── MCP Server 工具定义: ~10-20K  ← 最大隐形杀手！         │
│  └── LSP 状态: ~2-5K                                         │
├─────────────────────────────────────────────────────────────┤
│  半固定 (~5-10K)                                            │
│  ├── CLAUDE.md: ~2-5K                                       │
│  └── Memory: ~1-2K                                          │
├─────────────────────────────────────────────────────────────┤
│  动态可用 (~160-180K)                   ← 真正能用的只有这些 │
│  ├── 对话历史                                                │
│  ├── 文件内容                                                │
│  └── 工具调用结果                                            │
└─────────────────────────────────────────────────────────────┘
```

一个典型 MCP Server（比如 GitHub）包含 20-30 个工具定义，每个约 200 tokens。

**接 5 个 Server，光工具定义就要 25,000 tokens（12.5%）！**


### 分层加载策略：偶尔用的东西，别每次都加载

```
┌──────────────────┬────────────────────────────────────────┐
│  始终常驻         │  CLAUDE.md：项目契约/构建命令/禁止事项  │
├──────────────────┼────────────────────────────────────────┤
│  按路径加载       │  rules：语言/目录/文件类型特定规则       │
├──────────────────┼────────────────────────────────────────┤
│  按需加载         │  Skills：工作流/领域知识                │
├──────────────────┼────────────────────────────────────────┤
│  隔离加载         │  Subagents：大量探索/并行研究           │
├──────────────────┼────────────────────────────────────────┤
│  不进上下文       │  Hooks：确定性脚本/审计/阻断            │
└──────────────────┴────────────────────────────────────────┘
```

### 实用建议

- 保持 CLAUDE.md **短、硬、可执行**（Anthropic 官方自己的 CLAUDE.md 只有 2.5K tokens）
- 把大型参考文档拆到 Skills 的 supporting files
- 使用 `.claude/rules/` 做路径/语言规则
- 长会话主动用 `/context` 观察消耗

## 上下文压缩：别让算法把重要决策扔了

默认压缩算法按"可重新读取"判断，早期的 Tool Output 和文件内容会被优先删掉，**顺带把架构决策和约束理由也一起扔了**。

两小时后再改，可能根本不记得两小时前定了什么，莫名其妙的 Bug 就是这么来的 😅

### 解决方案一：Compact Instructions

在 CLAUDE.md 里写明压缩时必须保留什么：

```markdown
## Compact Instructions
When compressing, preserve in priority order:
1. Architecture decisions (NEVER summarize)
2. Modified files and their key changes
3. Current verification status (pass/fail)
4. Open TODOs and rollback notes
5. Tool outputs (can delete, keep pass/fail only)
```

### 解决方案二：HANDOFF.md

更主动的方案：开新会话前，先让 Claude 写一份 HANDOFF.md：

```
┌─────────────────────────────────────────────────────────┐
│  HANDOFF.md                                             │
├─────────────────────────────────────────────────────────┤
│  当前进度:                                               │
│  ├── 已完成: 用户认证模块重构                            │
│  ├── 进行中: 支付接口对接                                │
│  └── 待处理: 日志系统集成                                │
│                                                         │
│  尝试过的方案:                                           │
│  ├── ✅ JWT token 刷新机制 → 已验证可行                  │
│  ├── ❌ Redis 缓存方案 → 与现有架构冲突，已放弃          │
│  └── ✅ 数据库连接池 → 性能提升 40%                      │
│                                                         │
│  下一步:                                                │
│  └── 优先完成支付接口，再处理日志系统                    │
└─────────────────────────────────────────────────────────┘
```

下一个 Claude 实例只读这个文件就能接着做，不依赖压缩算法的摘要质量。

## Plan Mode：探索和执行，别混在一起

Plan Mode 的核心是把**探索和执行拆开**：

```
┌────────────────┐              ┌────────────────┐
│  探索阶段      │   确认方案   │  执行阶段      │
│                │ ──────────── │                │
│  • 只读操作    │              │  • 修改文件    │
│  • 澄清目标    │              │  • 运行测试    │
│  • 制定计划    │              │  • 验证结果    │
└────────────────┘              └────────────────┘
```

对于复杂重构、迁移、跨模块改动，这样做比"急着出代码"有用得多。它能显著降低在错误假设上持续修改的概率。

### 进阶玩法：让 AI 审 AI

开一个 Claude 写计划，再开一个以"高级工程师"身份审这个计划：

```
┌─────────────────┐         ┌─────────────────┐
│  Claude A       │  计划   │  Claude B       │
│  (编写计划)     │ ──────→ │  (审查计划)     │
│                 │ ←────── │  (高级工程师)   │
│  修改方案       │  反馈   │                 │
└─────────────────┘         └─────────────────┘
```

## Skills 的设计原则

### 官方定义

按需加载的知识与工作流。描述符常驻上下文，完整内容按需加载。

### 核心原则：Progressive Disclosure（渐进式披露）

不是让模型一次性看到所有信息，而是先获得索引和导航，再按需拉取细节：

```
┌─────────────────────────────────────────────────────────┐
│  SKILL.md                                               │
│  ├── 任务语义: 什么时候用这个 skill                     │
│  ├── 边界: 能做什么，不能做什么                          │
│  └── 执行骨架: 核心步骤概要                              │
│       ↓                                                 │
│  Supporting Files (按需加载)                             │
│  ├── runbook.md: 详细操作步骤                            │
│  ├── examples.md: 实际案例                               │
│  └── scripts/: 确定性收集脚本                            │
└─────────────────────────────────────────────────────────┘
```

### 一个稳定的 Skill 结构

```
.claude/skills/
└── incident-triage/
    ├── SKILL.md          # 任务语义、边界和执行骨架
    ├── runbook.md        # 领域细节
    ├── examples.md       # 示例
    └── scripts/
        └── collect-context.sh  # 脚本负责确定性收集
```

### 三种实用 Skill 类型

```
┌──────────────────┬────────────────────────────────────────┐
│  检查清单型       │  质量门禁：发布前跑一遍，确保不漏项      │
├──────────────────┼────────────────────────────────────────┤
│  工作流型         │  标准化操作：显式调用 + 内置回滚步骤    │
├──────────────────┼────────────────────────────────────────┤
│  领域专家型       │  封装决策框架：按固定路径收集证据        │
└──────────────────┴────────────────────────────────────────┘
```

### 描述符优化：每个 Skill 都在偷你的上下文空间

```yaml
# 低效（~45 tokens）
description: |
  This skill helps you review code changes in Rust projects.
  It checks for common issues like unsafe code, error handling...
  Use this when you want to ensure code quality before merging.

# 高效（~9 tokens）
description: Use for PR reviews with focus on correctness.
```

### 使用频率策略

```
┌─────────────────┬─────────────────────────────────────────┐
│  高频 (>1次/会话) │  保持 auto-invoke，优化描述符            │
├─────────────────┼─────────────────────────────────────────┤
│  低频 (<1次/会话) │  disable-auto-invoke，手动触发           │
├─────────────────┼─────────────────────────────────────────┤
│  极低频 (<1次/月)│  移除 Skill，改为文档                    │
└─────────────────┴─────────────────────────────────────────┘
```

## 工具设计的演进：给 agent 用 ≠ 给人用

给 Claude 的工具和给人写的 API 不是一回事。给 agent 用的重点不是功能堆得多完整，而是**让它更容易用对**。

### AskUserQuestion 的演进

Claude Code 团队前后试了三种做法：

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  第一版: 加参数                                             │
│  ├── 给 Bash 工具加 question 参数                           │
│  └── 结果: Claude 大多数时候直接忽略，根本不停下来问 ❌     │
│                                                             │
│  第二版: 约定输出格式                                       │
│  ├── 要求输出特定 markdown 格式                             │
│  └── 结果: Claude 经常"忘了"按格式写，逻辑很脆弱 ❌         │
│                                                             │
│  第三版: 独立工具                                           │
│  ├── 做成 AskUserQuestion 工具                              │
│  └── 结果: Claude 想提问就必须调用，调用即暂停 ✅           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

既然就是要 Claude 停下来问一句，那就直接给它一个专门的工具。加个 flag 或者约定一段输出格式，很多时候它一顺手就略过去了 🤷

## Hooks 的真正作用：收回确定性流程

Hooks 很容易被当成"自动运行的脚本"，但它更像是**把不能交给 Claude 临场发挥的事情，重新收回到确定性的流程里**。

比如格式化要不要跑、保护文件能不能改、任务完成后要不要通知，这些事真不要指望 Claude 每次都自己记得。

```
┌─────────────────────────────────────────────────────────┐
│  适合 Hooks:                                            │
│  ├── 阻断修改受保护文件                                  │
│  ├── Edit 后自动格式化/lint                              │
│  ├── SessionStart 后注入动态上下文                       │
│  └── 任务完成后推送通知                                  │
├─────────────────────────────────────────────────────────┤
│  不适合 Hooks:                                          │
│  ├── 需要读大量上下文的复杂语义判断                      │
│  ├── 长时间运行的业务流程                                │
│  └── 需要多步推理和权衡的决策                            │
└─────────────────────────────────────────────────────────┘
```

### 三层叠加防御

```
┌─────────────────────────────────────────────────────────┐
│  CLAUDE.md                                               │
│  └── 声明: "提交前必须通过测试和 lint"                   │
│       ↓                                                 │
│  Skill                                                   │
│  └── 告诉 Claude 运行测试的顺序、如何看失败、如何修复    │
│       ↓                                                 │
│  Hook                                                    │
│  └── 对关键路径执行硬性校验，必要时阻断                   │
└─────────────────────────────────────────────────────────┘
```

三样少任何一层都会有漏洞，放在一起才比较稳。

## Subagents 的核心价值：隔离不是并行

Subagent 的核心价值不是"并行"，而是**隔离**。

```
┌─────────────────┐         ┌─────────────────┐
│  主线程          │         │  Subagent       │
│                 │         │                 │
│  • 干净的上下文  │  派遣   │  • 独立上下文   │
│  • 只拿摘要     │ ←────── │  • 大量探索     │
│  • 不被污染     │  结果   │  • 隔离输出     │
└─────────────────┘         └─────────────────┘
```

扫代码库、跑测试、做审查这类会产生大量输出的事，交给 Subagent 做，主线程只拿摘要。

### 配置要点

```
┌─────────────────────┬─────────────────────────────────────┐
│  tools/限制工具      │  别给和主线程一样宽的权限            │
├─────────────────────┼─────────────────────────────────────┤
│  model 选择          │  探索用 Haiku/Sonnet，审查用 Opus   │
├─────────────────────┼─────────────────────────────────────┤
│  maxTurns           │  防止跑飞                          │
├─────────────────────┼─────────────────────────────────────┤
│  isolation: worktree │  需要动文件时隔离文件系统           │
└─────────────────────┴─────────────────────────────────────┘
```

## Prompt Caching：Cache Rules Everything

工程界有句话"Cache Rules Everything Around Me"，对 agent 同样如此。

### Prompt 缓存按前缀匹配工作

```
┌─────────────────────────────────────────────────────────┐
│  1. System Prompt          → 静态，锁定 ✓                │
│  2. Tool Definitions       → 静态，锁定 ✓                │
│  3. Chat History           → 动态，在后面                │
│  4. 当前用户输入           → 最后                        │
└─────────────────────────────────────────────────────────┘
```

### 破坏缓存的常见陷阱

```
┌─────────────────────────────────────────────────────────┐
│  ❌ 在静态系统 Prompt 中放入带时间戳的内容               │
│  ❌ 非确定性地打乱工具定义顺序                           │
│  ❌ 会话中途增删工具                                    │
└─────────────────────────────────────────────────────────┘
```

像当前时间这种动态信息怎么办？别去动系统 Prompt，放到下一条消息里传进去就行。

## 验证的重要性："Claude 说完成了"没啥用

"Claude 说完成了"其实没啥用，你得能知道它做没做对、出了问题能退回来、过程还能查。

```
┌─────────────────────────────────────────────────────────┐
│  更高层: 生产日志验证、监控指标、人工审查清单             │
│       ↑                                                 │
│  中间层: 集成测试、截图对比、contract test、smoke test    │
│       ↑                                                 │
│  最低层: 命令退出码、lint、typecheck、unit test          │
└─────────────────────────────────────────────────────────┘
```

在 Prompt、Skill 和 CLAUDE.md 中显式定义验证标准。

### 核心心得

> 假如一个任务你说不清楚"什么叫做完"，那大概率也不适合直接扔给 Claude 自动完成。验证标准本身都没有，Claude 再聪明也跑不出正确答案。

## CLAUDE.md 的编写指南

CLAUDE.md 更像是你和 Claude 之间的**协作契约**，不是团队文档，也不是知识库。

```
┌─────────────────────────────────────────────────────────┐
│  应该放什么:                                            │
│  ├── 怎么 build、怎么 test、怎么跑（最核心）             │
│  ├── 关键目录结构与模块边界                              │
│  ├── 代码风格和命名约束                                  │
│  ├── 那些不明显的环境坑                                  │
│  ├── 绝对不能干的事（NEVER 列表）                        │
│  └── 压缩时必须保留的信息                                │
├─────────────────────────────────────────────────────────┤
│  不应该放什么:                                          │
│  ├── 大段背景介绍                                        │
│  ├── 完整 API 文档                                       │
│  ├── 空泛原则，如"写高质量代码"                          │
│  ├── Claude 通过读仓库即可推断的显然信息                 │
│  └── 大量背景资料和低频任务知识                          │
└─────────────────────────────────────────────────────────┘
```

### 实用技巧

每次纠正 Claude 的错误后，让它自己更新 CLAUDE.md：

> "Update your CLAUDE.md so you don't make that mistake again."

## 总结

Claude Code 不是简单的 AI 助手，而是一个需要精心设计和治理的复杂系统。理解它的架构、合理组织上下文、正确使用各种工具，才能真正发挥它的威力。

核心一句话：**假如一个任务你说不清楚"什么叫做完"，那大概率也不适合直接扔给 Claude 自动完成。**
